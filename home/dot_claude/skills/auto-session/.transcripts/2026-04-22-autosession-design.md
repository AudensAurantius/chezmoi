## 1. Interactivity lever — yes, but model it as a small enum, not a boolean

`--headless` is really the end of a spectrum. I'd argue for three named tiers, because each collapses several independent switches into a coherent policy:

| Tier | Notifications | Decision blocks | Coordinator process | bead mutation policy | Permission-prompt behavior |
|---|---|---|---|---|---|
| `interactive` (default) | desktop toast + nag | halt + persistent toast | `claude` (TTY) | `default.yaml` | `ask` (halts) |
| `attended` | desktop toast + phone push | halt + phone push | `claude` (TTY) | `default.yaml` | `ask` |
| `headless` | phone push only | **log-and-proceed** with `revert_on_user_reject=true` | `claude -p` + `--output-format=stream-json`, no TTY | `headless.yaml` (stricter: `bd create` off, feature branches forced) | `bypassPermissions` with curated allowlist |

Key point: **headless mode can't use `Notification`/permission-prompt halts at all** — there's nobody at the terminal. The `decision_block` category has to flip from "halt coordinator" to "log + continue". Reversion plans in `decisions.md` become load-bearing because they're the only way to unwind a wrong call. That's a policy shift the scaffolder should enforce, not something the user toggles piecemeal.

Practical launcher detail: CC's `-p`/print mode is the real headless entry point. That expects a single prompt on the CLI and streams JSON. You don't pipe to an interactive `claude`. So `run.sh` for headless becomes a genuinely different shape — `exec claude -p "$(cat prompt.md)" --permission-mode bypassPermissions --output-format stream-json --session-id "$AUTOSESSION_SLUG"` or similar. Might be cleanest to generate `run.sh` vs `run-headless.sh` at scaffold time rather than branching at runtime.

## 2. ntfy alternatives

Straight comparison, since you asked:

- **ntfy.sh** — free, OSS, self-hostable, Android/iOS apps. Action-button support. Topic-based, zero-auth publish. Best "just works" option. Weakness: no native text-reply UI in the app (see #3).
- **Pushover** — paid one-time (~$5), extremely reliable, rich toasts, no reply path. Great for notify-only. Overkill if you already have ntfy working.
- **Gotify** — self-hosted ntfy-equivalent. Extra operational surface for no obvious gain over public ntfy unless you need privacy. Skip.
- **Telegram Bot API** — free, bidirectional by design, inline-keyboard callbacks, rich formatting, native reply textbox. Heavier setup (bot token, chat ID), but the reply story is *vastly* better than ntfy's. If "remote tether with text reply" is the goal, Telegram is the right tool.
- **Apprise** (library) — meta-dispatcher over ntfy, Pushover, Telegram, Slack, Discord, Signal, ~90 more. Single URL-like DSL (`ntfy://topic`, `tgram://bot/chat`). Doesn't replace a backend; it replaces *hand-rolling* the HTTP calls for each.

**My recommendation:** back the notifier with Apprise, not any single service. Dispatcher becomes `apprise -b "$body" -t "$title" -u urgent "${APPRISE_URLS[@]}"`. User configures a list of URLs in the alerts YAML; the rest is invariant. ntfy stays the easy default; Telegram is one config-line away if/when the user wants real replies.

## 3. Is ntfy bidirectional?

**Technically yes, practically awkward.** Breakdown:

- The topic itself is bidirectional — anyone who can publish *or* subscribe can do both. A phone that receives on `claude-$SLUG` can also POST to `claude-$SLUG`, or (cleaner) to `claude-$SLUG-reply`.
- The ntfy Android app doesn't have a native "type a reply" textbox on a notification. What it *does* have:
  - **Action buttons** (up to 3 per notification) that fire an HTTP request to any URL — so "Approve / Reject / Pause" style canned replies work out of the box.
  - **View actions** that deep-link you to the ntfy web UI, where you can publish to any topic manually.
- For freeform text replies from the phone, you're either: (a) installing Tasker/Shortcuts/KDE Connect to open an HTTP POST, or (b) asking the user to type in the ntfy web UI. Neither is great.

For **freeform text replies specifically, Telegram wins** — native textbox reply, bot receives via long-poll/webhook, done. So if bidirectional text is a key requirement, lean on Telegram for that channel and keep ntfy for fire-and-forget pushes.

Regardless of channel, the **inbound side is what's hard, not the outbound side**. Getting a reply from the phone to the coordinator requires a process that:
1. Listens (ntfy subscribe / Telegram long-poll / webhook)
2. Writes into the session — somehow. CC doesn't have an "inject text into a running session" input. Options:
   - Write to a file `<session-dir>/inbox.md`; coordinator polls at loop boundaries. **This is the only reliable option with stock CC.**
   - Use `PushNotification` tool to nudge the coordinator out of a wait state. Might not even work for headless. Uncertain.
   - Stdin injection via a wrapper (see #14 below) — hacky.

Pick the inbox file approach. It's simple, auditable, works for both interactive and headless, and survives compaction.

## 4. Composable hooks — yes, CC's schema supports this natively

Confirmed from your own `hooks/settings-fragment.json`: each event has an *array* of `{matcher, hooks}` entries, and each `hooks` entry is itself an *array* of `{type, command}`. Two levels of multiplicity. You can register 10 commands for `Notification` if you want; CC runs them all.

So the architectural question isn't "can we?" — it's "should multiplexing happen at the CC config layer or in a single dispatcher?"

**My strong recommendation: one hook per event, dispatch from there.** Reasons:

- Settings.local.json is declarative config; adding/removing notifiers means editing JSON, which is annoying and error-prone compared to toggling a YAML line.
- Dispatch order, failure isolation, and per-category routing are policy concerns best kept in one place, not split across N hook entries.
- The wrapper/watchdog pattern applies cleanly to one dispatcher. If you register N commands, you either wrap each (N watchdogs, N log entries per event) or wrap none.
- When you want a compact-responder + a nag + a phone push all on `Notification`, the routing ("is this an io_block or an idle prompt?") has to live *somewhere*. Putting it in the dispatcher is cleaner than relying on CC to run multiple hooks and having each independently decide to fire or not.

Concretely: keep `notify-dispatcher.sh` as the single Notification handler; extend it to read `<session-dir>/hooks.yaml` (or the unified session config) and fan out to an Apprise invocation, a wsl-notify-send, and a nag-spawn in parallel.

## 4b. User-specified hook executables + the wrapper question

> now that we're using project-localized hook configs, is this [the wrapper] even necessary?

Session-scoping and wrapping solve **orthogonal problems**:

- Session-scoping: this hook only fires when cwd is the session dir. Isolation.
- Wrapping: watchdog timeout, paired start/end log records, stderr capture, exit-code classification. Observability + resilience.

You still want wrapping even with session scope — a hung user-supplied hook will hang the session just as badly as a hung vendor hook. The two-record audit trail is the only way you'll ever debug a subtly-broken user hook.

**Do allow arbitrary executables as hook targets.** The mechanism is already in place — the scaffolder just needs to resolve `command:` paths at scaffold time and emit wrapped invocations. Good ergonomics: in the unified config, let the user write `command: ~/bin/my-notifier.sh` or `command: apprise://telegram/…` (dispatcher pattern) and have the scaffolder do the right thing.

## 5. scripts/ whitelist with symlink shims — no, skip it

You asked directly if this buys anything. Honest answer: **very little, and it adds a moving part you'll have to maintain.**

- **Safety**: ~zero. A symlink at `scripts/rm -> /usr/bin/rm` doesn't change what `rm` does. The permission grant `Bash($(pwd)/scripts/*)` just means "allow invoking anything in scripts/". That's not containment; it's aliasing.
- **Convenience**: marginal. You could just write `Bash(rm:*)` or a list of explicit patterns and get the same effect with less ceremony.
- **Auditability**: minor positive. An `ls -la scripts/` at a glance shows the allowed surface. But the permissions YAML already shows that.
- **Failure modes**: new ones introduced. Symlinks drift (target moved/renamed), break on chezmoi re-apply, behave subtly differently across filesystems.

The one place the pattern has merit is the **audit-wrapper** idea from #8 — there, the shim isn't a symlink, it's a real script that logs and execs. That's worth doing for its own reasons. Don't conflate the two.

## 6. `git worktree` vs clone — strong endorsement of worktree as the default

This is the most consequential architectural suggestion in your message. You're right to question the clone pattern.

Comparison:

| Aspect | `git worktree` | `clone` |
|---|---|---|
| Disk usage | Shared .git; O(working tree) | Full .git duplicate; O(working tree + history) |
| Setup speed | Instant | Slow on large repos |
| Push target | Real origin | Filesystem origin (then re-push) |
| History visibility | Unified — canonical sees the branch | Disjoint until push-back |
| Fetch/pull coordination | None needed; single .git | Two .git dirs can drift |
| Branch constraint | Can't checkout a branch already live in another worktree | No constraint |
| Accidental canonical-damage risk | Low — you're in a sibling dir but different index | Zero — physically separate |

For the chezmoi case specifically: `git worktree add ../chezmoi-autosession-feature feature/auto-…` gives you a fully isolated working tree, commits go to a separate branch, `git push origin feature/…` pushes straight to the real remote. No need for "push back to filesystem origin then push again." The canonical's checked-out main is untouched.

**The one case clone still wins:** you want a different `remote.origin.url` (pushing to a fork, or simulating a migration). You'll know when you need it.

Recommendation: make `worktree` the default location value, keep `clone` as opt-in with a documented rationale ("use when you need disjoint remotes or are testing repo-destructive operations"), and update `templates/repos/chezmoi.md` to pin `worktree+feature` rather than `clone+feature`. The scaffolder logic only needs minor changes — instead of `git clone` invoke `git worktree add` and record the worktree path in state.json so shutdown can remove it.

Note on cleanup: worktrees need `git worktree remove` or `git worktree prune` — leaving orphans in `.git/worktrees/` is sloppy. Add this to the shutdown checklist.

## 7. Audit-only permissions — yes, this is a good idea

CC's permission schema is `allow | deny | ask` — no `audit` tier. But you can synthesize one:

1. User declares `audit_patterns: [rm, mv, git push]` (or similar) in the session config.
2. Scaffolder generates a shim per pattern at `<session-dir>/scripts/audit/<cmd>`:
   ```sh
   #!/usr/bin/env bash
   # Auto-generated audit shim for `rm`.
   echo "{\"ts\":\"$(date -Iseconds)\",\"cmd\":\"rm\",\"args\":$(printf '%s\n' "$@" | jq -Rs .)}" >> "$AUTOSESSION_DIR/audit.jsonl"
   exec /usr/bin/rm "$@"
   ```
3. Permissions grant `Bash($(pwd)/scripts/audit/*)` and deny the bare patterns (`Bash(rm:*)` → deny).
4. Coordinator must use `./scripts/audit/rm` to run rm — any direct `rm` is denied.

This gives you **"allowed but audited"** as a first-class concept, backed by real mechanism. Cost: the coordinator has to resolve audit commands through the shim path, which requires a prompt-level instruction ("For any audited command, invoke via `./scripts/audit/<cmd>` not directly"). That's where it gets iffy — relying on LLM discipline to route through the shim isn't bulletproof.

Better variant: use `PreToolUse` hook as the actual audit mechanism (it fires on every tool call including Bash, sees the full command line, can log + optionally deny). No shim farm needed; the hook is the audit layer. The session's Bash permissions stay `allow`, the `PreToolUse` hook logs matching patterns to `audit.jsonl`.

I'd go with **the hook approach**: no shim generation, no dependency on coordinator discipline, and the hook fires deterministically every call. The shim approach is a valid fallback if you ever need to audit a non-Bash tool class (but you don't — see #9).

## 8. Wrap every tool call — mostly won't work, hook it instead

Claude Code's non-Bash tools (Read, Edit, Grep, Glob, Write, Agent, etc.) are executed *inside the CC client process*. There's no user-specifiable exec path for them. You can't "wrap Read with a watchdog script." The only tool that shells out to user code is Bash.

What you *can* do is hook all tool calls via `PreToolUse` / `PostToolUse`. Those fire for every tool, receive the tool name and inputs as JSON on stdin, and can block or allow. That IS the "wrap every tool call" primitive, just expressed as a hook event rather than a shell wrapper.

So combine #7 and #8 into: **use `PreToolUse` + `PostToolUse` as a single audit event stream.** The hook writes JSONL records of (tool, inputs, decision, timing) into `tool-audit.jsonl`. You already have the wrapper/watchdog infrastructure for hooks — this just adds a handler.

## 9. Wrap Claude Code itself — wrong mechanism for the stated goal

Your stated goal ("pipe input to bypass user-interaction events for full autonomy") is real, but the mechanism isn't "wrap stdin of an interactive claude." The mechanism is **headless mode from #1**:

- `claude -p "..."` with `--permission-mode bypassPermissions` (or `acceptEdits`) — documented modes for non-interactive execution.
- The SDK `@anthropic-ai/claude-agent-sdk` for programmatic session construction.

If you wrap interactive `claude` and try to detect when it's reading stdin and inject responses, you're reimplementing the SDK poorly. You also lose structured output (events, decisions) unless you re-parse the TUI.

The clean split: `interactive` mode uses `exec claude` in the TTY; `headless` mode uses `claude -p` with stream-json output that a wrapper can parse to emit your events.jsonl records directly — no guessing from TUI strings.

## 10. User-specified logging command — yes, trivial and useful

The wrapper already has a single sink (`$LOGFILE`). Generalize it to an executable or shell pipeline: `AUTOSESSION_LOG_SINK="$HOME/bin/log-to-sqlite.sh"`, wrapper pipes each JSON record into it on stdin. Default falls back to file-append.

Same pattern for `events.jsonl`: the coordinator currently appends to a file; let it append to a configurable command that can be a file, a SQLite inserter, an OpenTelemetry exporter, whatever. The event schema is already stable enough to commit to as the contract.

Low cost, high value. The example you gave (flatten to SQLite records) would become the project's reference sink — consider shipping it in `scripts/sinks/sqlite.sh`.

## 11. Autosession-scoped beads sandbox — YAGNI

You already have:

- A bead mutation policy (`templates/beads/*.yaml`) that constrains the coordinator.
- Transient labels (`autosession-deferred`, `autosession-blocked`) removed on next claim.
- Cardinality-1 metadata (`agent=claude`, `autosession_slug=…`) for classification.
- Prefixed notes (`[autosession/<slug> <ISO>]`) for an auditable transition trail.

The problem a sandbox DB would solve: coordinator experiments with bead graph manipulation, formulas, molecules that shouldn't outlive the session. In practice, your `bd create` is *disabled* in the default policy — the coordinator isn't creating beads at all, let alone exploratory ones. The policy already handles the risk.

Adding a sandbox DB means:

- Sync problem (import beads from canonical → sandbox; push closures back).
- `bd ready`/`bd close` retargeting — every command needs `--db` or an env var.
- Two sources of truth for a brief window, then merge.
- Extra discipline for the coordinator (twice the surface for mutation policy to cover).

For the benefit of… letting the coordinator create ephemeral planning beads? Scratchpad files in `<session-dir>/scratchpad/goals/` already do that, with zero sync cost.

**Ship without it. Revisit if the pattern shows up organically** (e.g., you find yourself wanting to run `bd formula` / `bd mol` within a session and have those artifacts be scoped). At that point, you have concrete requirements instead of a speculative design.
